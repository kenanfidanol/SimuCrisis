SimuCrisis – Analysis Report

Group 46

1. Introduction

SimuCrisis is a pandemic simulation program. The system is designed to provide educational features, such as helping users understand how different management strategies affect the spread of a virus. SimuCrisis models infections and recoveries using historical datasets.

The system aims to provide real-time visual feedback through charts, heatmaps, and scenario reports. SimuCrisis is intended for educators, researchers, students, and policymakers who want to explore extreme pandemic scenarios and ways to mitigate them in a safe, controlled digital environment.

Initially, the project considered VR support, but after evaluation, VR was removed due to limited educational added value and accessibility concerns. The system now focuses entirely on a desktop Unity application.

2. Current System

There is no existing system that SimuCrisis is built upon. All components are being developed from scratch.

2.1 Data Input System

  1. User (Actor)
    • The User interacts with the system by selecting datasets, loading predefined scenarios, or entering simulation parameters (infection rate, recovery time, population size, etc.).
  2. Data Input Interface (Component)
    • Collects numerical and categorical inputs from the User.
    • Validates user input (value ranges, data type consistency).
    • Sends the validated input to the Simulation Core.
  3. Dataset Module (Component)
    • Stores historical pandemic datasets (e.g., COVID-19, H1N1).
    • Provides structured data to the Data Input Interface.
    • Validates dataset format and readiness for simulation.

Process Flow:

  1. User selects a dataset or enters custom parameters.
  2. Data Input Interface validates all inputs.
  3. Dataset Module loads relevant files or values.
  4. Validated data is passed to the Simulation Core to begin the simulation run.

Arrows and Communication:

  • Arrows show the flow of parameter data from User → Interface → Dataset Module → Simulation Core.
  • Dashed lines indicate repeated parameter adjustments before the simulation begins.

2.2 Population Simulation Processing

  1. User (Actor)
    • Initiates the simulation, pauses it, changes the speed, or adjusts parameters during runtime.
  2. Simulation Core (Component)
    • Implements the epidemiological model (e.g., SEIR or a simplified variant).
    • Calculates transitions between susceptible, exposed, infected, and recovered groups.
    • Updates simulation state at each timestep.
    • Processes population interactions and transmission probabilities.
  3. Population Data Module (Component)
    • Stores all demographic and synthetic population characteristics.
    • Supplies the Simulation Core with structured population data.

Process Flow:

  1. User starts the simulation.
  2. Simulation Core initializes the population state.
  3. Population Data Module provides demographic structures.
  4. Simulation Core updates infection, recovery, and fatality events at each timestep.

Arrows and Communication:

  • Continuous data stream from Population Data Module to Simulation Core.
  • Dashed lines represent repeated update cycles during simulation runtime.

2.3 Visualization Processing

  1. User (Actor)
    • Observes charts, heatmaps, and graphs during the simulation.
  2. Visualization Engine (Component)
    • Converts simulation state into visual elements.
    • Updates graphs in real time (infection rates, reproduction number R, hospital load).
    • Generates spatial heatmaps of population spread.
  3. UI Layer (Component)
    • Displays all visual output to the User.
    • Handles user commands such as zooming, switching graph types, or toggling layers.

Process Flow:

  1. Simulation Core sends the updated state to the Visualization Engine.
  2. Visualization Engine processes data into graphical objects.
  3. UI Layer renders these visuals and updates the screen.
  4. User interacts with the UI to change visualization settings.

Arrows and Communication:

  • Solid lines represent real-time data transfer for visual updates.
  • Dashed lines indicate repeated interactions from the User to adjust views.

2.4 Policy Application Processing

  1. User (Actor)
    • Applies interventions such as lockdowns, school closures, vaccination campaigns, and healthcare capacity expansion.
  2. Policy Module (Component)
    • Evaluates each intervention.
    • Calculates how the intervention modifies simulation parameters (e.g., reduced contact rate).
  3. Simulation Core (Component)
    • Integrates modified parameters into the ongoing simulation.

Process Flow:

  1. User selects a policy to apply.
  2. Policy Module evaluates the policy impact.
  3. Updated parameters are sent to the Simulation Core.
  4. Simulation Core continues with adjusted dynamics.

Arrows and Communication:

  • Arrows represent communication from interventions to the Simulation Core.
  • Dashed arrows indicate repeated policy adjustments.

2.5 Scenario Comparison Processing

  1. User (Actor)
    • Selects two or more completed simulation runs to compare.
  2. Scenario Manager (Component)
    • Loads stored scenario data.
    • Extracts metrics such as peak infections, duration, death toll, and healthcare overload.
  3. Comparison Engine (Component)
    • Generates visual comparisons (side-by-side graphs, difference plots).
    • Highlights differences in policy effectiveness between scenarios.

Process Flow:

  1. User selects scenarios to compare.
  2. Scenario Manager loads stored data.
  3. Comparison Engine generates visual outputs.
  4. Results are displayed to the User.

Arrows and Communication:

  • Dashed lines show repeated comparisons across multiple scenarios.

2.6 Reporting and Output Processing

  1. User (Actor)
    • Requests a simulation summary report.
  2. Report Generator (Component)
    • Collects final simulation statistics.
    • Formats them into exportable PDF/CSV reports.
  3. Data Storage Module (Component)
    • Provides simulation logs and scenario metadata.

Process Flow:

  1. User requests a report.
  2. Report Generator gathers the necessary data.
  3. Data is formatted and exported in the selected format.
  4. The report is delivered to the User.

3. Proposed System

3.1 Overview

SimuCrisis will simulate the progression of infectious diseases in a virtual population and provide real-time visual analytics. Users can test interventions and explore outcomes. The system emphasizes accessibility, clarity, and educational value. VR functionality has been removed due to low practical impact.

3.2 Functional Requirements

  • Simulate infection, recovery, and death dynamics.
  • Support adjustable parameters for the disease model.
  • Apply interventions such as lockdowns, vaccination, and resource expansion.
  • Display real-time graphs and heatmaps.
  • Save and load simulation scenarios.
  • Compare scenarios visually.
  • Export summary reports.

3.3 Non-Functional Requirements

  • Performance: Real-time updates with minimal latency.
  • Reliability: Consistent calculations and robust error handling.
  • Usability: Simple user interface with easy navigation.
  • Scalability: Ability to support additional datasets and models.
  • Maintainability: Modular architecture to ease updates and extensions.

3.4 Pseudo Requirements

  • Hardware: Standard desktop PC capable of running Unity applications.
  • Software: Unity engine and publicly available datasets.
  • Network: Not required, except for optional dataset updates.
  • Training: Minimal training; user-friendly interface.
  • Maintenance: Occasional software updates.

4. Glossary

SEIR Model
A common epidemiological model that divides the population into Susceptible, Exposed, Infected, and Recovered groups.
R (Effective Reproduction Number)
A measure of how many people, on average, a single infected person will infect at a given time in the population.
Scenario
A saved simulation state that can be reloaded and compared with other runs.
Intervention
A user-applied strategy (e.g., lockdown, vaccination campaign, capacity expansion) that modifies simulation parameters.
Heatmap
A spatial distribution visualization showing intensity of infections across regions.
