1. Features to Be Tested

This section identifies the main functional and quality-related parts of SimuCrisis that will be tested. Because the project is an educational pandemic disaster management simulation, testing focuses on model correctness, policy-response behavior, usability, reproducibility, and the reliability of exported results.

1.2 Non-Functional Features

  • Correctness and numerical stability of the simplified SEIR model over repeated timestep execution.
  • Responsiveness of dashboard interactions when parameters or policy controls are changed.
  • Reproducibility of results when the same scenario settings and data inputs are reused.
  • Stability during long sessions, including repeated runs, resets, and scenario comparisons.

2. Testing Methodology

Testing in SimuCrisis will combine automated checks and manual review. Automated tests are most suitable for validating deterministic model calculations, save/load consistency, and export accuracy. Manual tests are necessary for dashboard clarity, scenario-based workflows, and the educational usefulness of the visualization outputs.

2.1 Unit Testing

  • Scope: core methods inside SimulationCore, PolicyEngine, MetricsCollector, ScenarioManager, and ReportGenerator.
  • Tools: Unity Test Framework for EditMode/PlayMode tests, plus debug logs for targeted validation.
  • Goal: verify method outputs, input validation, boundary conditions, and error handling.

2.2 Integration Testing

  • Scope: interactions between model logic, policy application, metrics calculation, dashboard updates, and export features.
  • Goal: confirm that data moves correctly between modules and that one module's output becomes another module's correct input.
  • Strategy: incremental integration, starting from logic-layer components and moving toward user-facing workflows.

2.3 System Testing

  • Environment: the same desktop platforms intended for demonstrations and evaluation.
  • Tests: end-to-end workflows such as selecting a scenario, changing policies, running the simulation, comparing outcomes, and exporting results.
  • Focus: correctness of the full workflow, stability of repeated use, and clarity of the educational outputs.

2.4 Performance Testing

  • Metrics: initialization time, update time, UI response time, export duration, and long-session stability.
  • Targets: thresholds are defined to keep the simulation usable during demonstrations, classroom review, and project evaluation.
  • Tools: Unity Profiler, application logs, stopwatch-based measurement for repeated runs, and OS-level resource observation when needed.

2.5 User Acceptance Testing (UAT)

  • Participants: selected students, peers, and academic reviewers who represent the intended educational audience.
  • Focus: ease of use, clarity of outputs, interpretability of graphs and heatmaps, and perceived learning value.
  • Tools: structured observation notes and a short Likert-scale feedback form.

2.6 Beta Testing

  • Scope: short pilot use outside the core development session, preferably on more than one machine and with more than one user.
  • Focus: uncovered bugs, platform-specific issues, and weaknesses that do not appear during controlled developer testing.
  • Duration: a compact 3-5 day feedback window is sufficient for the final project phase.

3. Test Method

This section explains how each testing activity will be performed in practice. The methods are selected to match the structure of SimuCrisis, where core logic is deterministic and easy to automate, while visual interpretation and educational usability require direct human observation.

3.1 Unit Testing

Objective: validate that each independent module performs its intended task correctly under normal, boundary, and invalid input conditions.

Application: the Simulation Core will be tested with controlled timestep inputs to confirm valid SEIR transitions and stable totals. The Policy Engine will be tested to confirm that a policy changes only the intended parameters. The Metrics Collector will be tested using known snapshots so that calculated outputs can be compared against expected values.

Method: tests will use small deterministic inputs whenever possible so that expected outputs can be checked exactly. Invalid inputs, such as missing scenario fields or out-of-range numerical values, will be included to verify graceful failure behavior.

3.2 Integration Testing

Objective: confirm that separate modules cooperate correctly once they are connected.

Application: scenario-loading tests will verify that stored parameters become the live model configuration without loss or corruption. Policy application tests will verify that the updated parameters influence the next valid simulation outputs and that the dashboard refreshes using the updated data rather than cached data.

Method: integration tests will be executed incrementally. Logic-layer integrations will be checked first, then UI-linked integrations, then export-linked integrations. This order helps isolate faults early and reduces debugging time.

3.3 System Testing

Objective: evaluate the full SimuCrisis workflow from a user point of view.

Application: complete workflows will include creating or loading a scenario, applying interventions, running multiple timesteps, reviewing charts and heatmaps, comparing policy outcomes, and exporting results. These tests aim to reveal faults that do not appear in module-level testing, such as broken navigation, stale data in the dashboard, or incomplete exports.

Method: realistic user stories will be followed step by step. Observed outputs will be compared against expected workflow behavior defined in the design documents and in the scenario configuration.

3.4 Performance Testing

Objective: measure responsiveness and resource behavior under realistic educational use conditions.

Application: repeated runs will be performed with standard and heavy scenarios, especially those with multiple policy changes, long timelines, and several active visual outputs. Startup time, chart refresh speed, export time, and overall stability will be measured.

Method: each performance case will be repeated multiple times on the same build. Outliers will be noted separately so that average behavior and worst-case behavior can both be discussed in the final report.

3.5 User Acceptance Testing (UAT)

Objective: determine whether the final system is understandable and useful for its intended audience.

Application: participants will be given short scenario tasks, such as adjusting interventions to reduce infection peak or comparing two strategies. They will then be asked whether the controls were understandable, whether the outputs were meaningful, and whether the system supported learning.

Method: the team will observe task completion, collect short written feedback, and convert feedback into actionable UI or explanation improvements when possible.

3.6 Beta Testing

Objective: validate the system in a less controlled environment than the core development setup.

Application: a small pilot version of SimuCrisis will be used by selected reviewers over several days. The aim is to detect platform-specific issues, unclear instructions, or unusual user actions that the development team may not anticipate.

Method: issues discovered during beta use will be logged, prioritized, fixed if time allows, and followed by focused regression checks on the affected workflow.

4. Test Environment

The test environment for SimuCrisis is designed to reflect the hardware and software conditions under which the project will be demonstrated and evaluated. Since the project is a Unity-based educational desktop simulation rather than a field-deployed medical system, the environment emphasizes reproducibility, desktop stability, and ease of observation.

4.1 Hardware Environment

  • Desktop or laptop computers used by the development team and academic reviewers.
  • Multi-core CPU and at least 16 GB RAM recommended for stable execution and profiling during demonstrations.
  • Standard 1080p or similar display resolution for evaluation of dashboard readability and chart layout.
  • Solid-state storage preferred to reduce load and save times for scenario and export testing.

4.2 Software Environment

  • Unity project environment (Unity 2022 LTS or the team's selected stable Unity version).
  • C# scripts implementing the simulation logic, UI control flow, visualization behavior, and export functions.
  • Operating systems: Windows and/or macOS machines used by the team and demonstration audience.
  • Structured input/output files such as JSON, CSV, and configuration assets used for scenarios and exports.
  • Unity Test Framework for automated logic tests, Unity Profiler for performance observation, and a shared issue tracker for defect follow-up.

4.3 Test Data and Realistic Simulation Conditions

  • Historical pandemic datasets and structured reference parameters used to initialize realistic learning scenarios.
  • Synthetic edge-case scenarios, such as extreme transmission rates, unrealistic capacity limits, or abrupt policy changes, used to stress the model safely.
  • Repeated runs of the same scenario configuration to verify reproducibility and consistency.
  • Scenario sets representing low, medium, and high outbreak severity so that the team can compare both normal and stress conditions.

6. Test Monitoring and Tracking Strategy

6.1 Test Result Tracking

Test execution results will be recorded in a shared tracking file or project board so that the full testing history remains visible to the team. Each test record should include at least the following fields:

  • Test ID and test type
  • Module or workflow tested
  • Build version / commit reference
  • Execution date and tester name
  • Pass / Fail / Blocked status
  • Observed result and short remarks
  • Linked defect ID if the test fails

6.2 Defect Management

Whenever a test fails, the issue will be logged in the shared defect tracker. The report must be detailed enough for another team member to reproduce the problem without guessing. A proper defect entry should include the issue title, reproduction steps, expected behavior, observed behavior, build information, and screenshots or logs where relevant.

Priority levels: Critical (blocks essential workflow), High (major feature affected), Medium (partial workaround exists), Low (minor clarity or cosmetic issue).

6.3 Bug Resolution Workflow

  1. A failed test or user report creates a defect entry in the shared tracker.
  2. The issue is assigned to the most relevant team member or to the first available developer if ownership is shared.
  3. The developer reproduces the issue, implements a fix, and updates the build.
  4. The affected test case is executed again to confirm the fix.
  5. Related regression checks are executed on nearby features to make sure the fix did not create a new problem.
  6. If the issue is resolved, the defect entry is closed and linked to the final test summary.

7. Roles and Responsibilities

Because SimuCrisis is developed by a small senior design team, testing responsibilities are shared. However, each testing cycle should still have clear coordination so that tasks are not duplicated or missed.

7.1 Test Coordinator / QA Lead

  • Prepares the weekly test focus, maintains the test list, and makes sure execution results are recorded.
  • Checks whether critical defects have owners and whether re-tests are completed before the next phase begins.
  • Summarizes major findings for team meetings and final reporting.

7.2 Developers / Feature Owners

  • Create unit tests for the modules they implement or modify.
  • Fix defects found during testing and provide enough technical detail for re-testing.
  • Perform quick self-checks before handing the build to the rest of the team.

7.3 Peer Testers / Review Participants

  • Execute manual test cases, especially workflow, usability, and interpretation tasks.
  • Report unclear behavior, misleading outputs, and usability pain points.
  • Participate in UAT and short beta sessions when requested.

7.4 Supervisor / Quality Gate

  • Reviews whether essential functionality, stability, and educational goals are satisfied.
  • Provides high-level feedback on whether the system is ready for presentation or submission.
  • Acts as a final checkpoint for acceptance of the project deliverable.

8. Risks in the Testing Process

  • Time Constraints: The testing window in a senior design project is limited. If major issues appear late, the team may have to prioritize only the most critical fixes.
  • Model Simplification Risk: Because the project uses a simplified SEIR-style model, some outputs may be educationally useful but still sensitive to parameter tuning. Tests must therefore verify internal consistency and intended behavior rather than claiming real-world epidemiological precision.
  • Data Quality Issues: Historical or reference datasets may contain missing values, inconsistent formats, or parameter assumptions that affect scenario initialization.
  • Integration Problems: A correct model layer can still produce incorrect visible results if the dashboard, charts, or export layer receives outdated or mismatched data.
  • Performance and Stability Problems: Long sessions, repeated comparisons, or large scenario histories may slow the system or expose memory-related issues.
  • Save / Load Failures: Scenario persistence errors could damage reproducibility, which is one of the core educational goals of the project.
  • Late User Feedback: If peer or reviewer feedback arrives late, UI and workflow improvements may not be fully incorporated before submission.
  • Cross-Platform Differences: Behavior on different Windows or macOS machines may reveal file path, layout, or export inconsistencies not visible on the main development machine.

9. Example Test Case List

The following sample cases show how concrete SimuCrisis tests can be documented. The list is not exhaustive; it is intended to represent the most important logic, workflow, usability, and performance checks.

TC-001: SEIR State Conservation Test

  • Test ID: TC-001
  • Feature to be Tested: The Simulation Core's timestep update behavior.
  • Input Data / Preconditions: A valid initialized scenario with known Susceptible, Exposed, Infected, and Recovered values.
  • Expected Output: After each timestep, no compartment becomes negative and the total population remains consistent.

TC-002: Invalid Parameter Validation Test

  • Test ID: TC-002
  • Feature to be Tested: Input validation for scenario parameters.
  • Input Data / Preconditions: A scenario with invalid values such as negative population, impossible rates, or missing required fields.
  • Expected Output: The system rejects the invalid configuration gracefully and provides a clear error or warning instead of crashing.

TC-003: Lockdown Policy Application Test

  • Test ID: TC-003
  • Feature to be Tested: The Policy Engine's handling of intervention changes.
  • Input Data / Preconditions: A running scenario where a lockdown policy is activated at a defined timestep.
  • Expected Output: The related transmission settings change at the correct time and the next updates reflect the policy effect.

TC-004: Vaccination Strategy Effect Test

  • Test ID: TC-004
  • Feature to be Tested: Dynamic policy application for vaccination-related settings.
  • Input Data / Preconditions: Two otherwise identical scenarios, one with vaccination intervention enabled and one without it.
  • Expected Output: The compared outputs show a meaningful difference in trend, and the dashboard labels the comparison clearly.

TC-005: Scenario Save / Load Consistency Test

  • Test ID: TC-005
  • Feature to be Tested: Scenario Manager persistence functions.
  • Input Data / Preconditions: A configured scenario with non-default intervention settings and metadata.
  • Expected Output: The reloaded scenario matches the saved settings exactly and produces the same initial simulation state.

TC-006: Visualization Refresh Test

  • Test ID: TC-006
  • Feature to be Tested: The connection between model state and charts / heatmaps.
  • Input Data / Preconditions: A scenario is running while the user changes a policy or advances several timesteps.
  • Expected Output: The displayed graphs and indicators refresh to the latest data without stale values, missing labels, or inconsistent scales.

TC-007: Historical Data Loading Test

  • Test ID: TC-007
  • Feature to be Tested: Historical Data Loader / Data Management initialization.
  • Input Data / Preconditions: A valid structured data file and a second malformed file for negative testing.
  • Expected Output: The valid file loads correctly; the malformed file is rejected with a controlled error message.

TC-008: End-to-End Workflow Test

  • Test ID: TC-008
  • Feature to be Tested: Full user workflow from scenario selection to export.
  • Input Data / Preconditions: A complete build with at least one prepared scenario and export target location.
  • Expected Output: The user can load a scenario, apply interventions, run the model, review outputs, and export a summary without blocking errors.

TC-009: Export Accuracy Test

  • Test ID: TC-009
  • Feature to be Tested: Report / Export Module accuracy.
  • Input Data / Preconditions: A finished simulation with known on-screen metrics.
  • Expected Output: Exported CSV or PDF values match the final displayed metrics and include the required scenario information.

TC-010: Long-Session Stability Test

  • Test ID: TC-010
  • Feature to be Tested: Stability under extended use.
  • Input Data / Preconditions: Repeated runs, resets, comparisons, and exports during a continuous 30-minute session.
  • Expected Output: The application remains responsive, does not crash, and does not show severe slowdown or corrupted outputs.
