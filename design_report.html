High Level Design Report

3. Proposed Software Architecture

3.1 Overview

The modular architecture of SimuCrisis allows the Simulation Core to maintain the epidemiological model state while broadcasting state snapshots to downstream components. User interactions are processed through the User Interface and translated into policy updates by the Policy Engine.

Time-series indicators such as the effective reproduction number (Rt), infection peaks, and healthcare load are computed by the Metrics Collector. The Visualization Engine renders charts and heatmaps based on these metrics. Scenario persistence and export operations are handled by the Scenario Manager and Report Generator subsystems.

3.2 Subsystem decomposition

  • UI Layer (Unity): Parameter Panel, Policy Panel, Dashboard (Charts/Heatmap), Scenario Comparison UI
  • Core Layer: Simulation Core (SEIR Model), Policy Engine, Metrics Collector, Visualization Engine
  • Data Layer: Scenario Manager, Report Generator
  • Local storage for scenarios, configuration metadata, and historical datasets

3.3 Hardware / software mapping

The system is designed to operate on a standard desktop computer and does not require specialized hardware such as virtual reality devices. The primary implementation environment is Unity using the C# programming language.

Optional Python tools may be used for offline data analysis or preprocessing, while all persistent data is stored locally using structured files.

3.4 Persistent data management

SimuCrisis stores simulation scenarios and run artifacts locally. These artifacts include selected dataset identifiers, simulation parameters, intervention schedules, random seeds used for reproducibility, and summary statistics generated during the simulation.

No personal or private health information is stored by the system.

3.5 Access control and security

The system operates locally and does not require user authentication in its initial version. Scenario files can be validated prior to loading to prevent corrupted or invalid inputs.

Future extensions involving online sharing or collaboration would require access control mechanisms and secure communication channels.

3.6 Global software control

SimuCrisis follows an event-driven control structure. User actions such as starting, pausing, or modifying simulation speed trigger events that update internal parameters.

Each simulation timestep generates a snapshot that is used to update metrics and refresh visualizations.

3.7 Boundary conditions

Simulation performance may degrade for very large population sizes or dense spatial representations. To mitigate this, the system applies bounded timestep sizes and aggregated compartment logic.

Parameter validation mechanisms prevent unstable configurations, such as negative transition rates or impossible healthcare capacities.

4. Subsystem Services

Simulation Core

Executes the SEIR epidemiological model and updates the state of the simulated population at each timestep based on current parameters.

Policy Engine

Applies user-defined interventions such as lockdowns or vaccination strategies and adjusts simulation parameters dynamically according to policy schedules.

Metrics Collector

Computes analytical indicators including the effective reproduction number (Rt), infection peaks, and healthcare utilization metrics.

Visualization Engine

Generates real-time charts and spatial heatmaps to visually represent infection dynamics.

Scenario Manager

Handles saving, loading, and comparison of simulation scenarios for reproducibility and analysis.

Report Generator

Exports simulation results in PDF and CSV formats for external reporting and evaluation.

6. Glossary

Heatmap:
A spatial visualization technique used to represent the intensity or concentration of a specific variable across a geographic or simulated space. In SimuCrisis, heatmaps are primarily used to display infection density and help users visually identify high-risk or heavily affected areas.

Scenario:
A stored simulation configuration that includes initial conditions, disease parameters, and a predefined set of intervention policies. Scenarios allow users to reproduce experiments, compare different strategies, and analyze outcomes under consistent starting assumptions.

Intervention:
A user-defined policy action applied during a simulation run to influence disease dynamics. Examples include lockdown measures, vaccination strategies, and healthcare capacity adjustments. Interventions modify simulation parameters in a controlled manner over time.

Snapshot:
A representation of the complete simulation state captured at a specific timestep. Snapshots include population distributions, calculated metrics, and relevant system variables, and are used for both real-time visualization and post-simulation analysis.
