2. Testing Methodology

Testing in SimuCrisis will combine automated checks and manual review. Automated tests are used mainly for validating deterministic model calculations, scenario consistency, and export accuracy. Manual tests are necessary for dashboard clarity, visualization readability, and the educational interpretation of simulation results.

2.1 Unit Testing

  • Scope: core simulation logic, policy application, metrics calculation, scenario management, and report generation.
  • Tools: Unity Test Framework for EditMode and PlayMode tests, plus lightweight debug logs.
  • Goal: ensure small modules behave independently and produce valid outputs under normal and edge-case conditions.

2.2 Integration Testing

  • Focus: interactions between model logic, policy application, metrics calculation, dashboard updates, and export features.
  • Goal: confirm that data moves correctly between modules and that no invalid or stale values appear after state transitions.

2.3 System Testing

  • Environment: the same desktop platform intended for demonstration and educational usage.
  • Testing: complete workflows such as loading a scenario, changing policies, running the simulation, comparing outcomes, and exporting reports.
  • Purpose: verification of the full workflow, stability of repeated runs, and clarity of the educational outputs.

2.4 Performance Testing

  • Metrics: initialization time, update time, UI response time, export duration, and long-session stability.
  • Targets: responsive behavior for large but realistic outbreak simulations, classroom sessions, and project evaluation demos.
  • Tools: Unity Profiler, debug logs, simple benchmark measurements of repeated runs, and GitHub issue tracking for observed slowdowns.

2.5 User Acceptance Testing (UAT)

  • Participants: selected students, peers, and academic reviewers who represent the intended educational audience.
  • Focus: ease of use, clarity of outputs, interpretability of graphs and heatmaps, and perceived learning value.
  • Tests: realistic educational scenarios and short classroom-feedback flows.

2.6 Beta Testing

  • Targets: short pilot execution on non-development machines, portability checks, and external reviewer feedback.
  • Purpose: uncover bugs, platform-specific issues, and workflow confusion not identified during controlled developer testing.

3. Test Methods

This section explains how the testing activity will be performed in practice. The methods are selected to match the structure of SimuCrisis while ensuring both correctness of the internal model behavior and educational usability during demonstration use.

3.1 Unit Testing

Objectives: validate that each independent module performs its intended task correctly under normal, boundary, and invalid input conditions.

Approach: the Simulation Core will be tested with controlled timestep inputs to confirm valid SEIR transitions and stable health calculations. The Policy Engine will be tested to confirm that policy changes only affect intended parameters. The Metrics Collector will be tested using known snapshots so that calculated outputs can be compared against expected values.

Validation: limits, null values, deterministic-repeat behavior, exported report structure, and scenario consistency checks.

4.3 Test Data and Realistic Simulation Conditions

  • Historical pandemic datasets and structured reference parameters used to initialize realistic learning scenarios.
  • Synthetic edge-case scenarios, such as extreme transmission rates, unrealistic capacity limits, or abrupt policy changes, used to stress the model safely.
  • Repeated runs of the same scenario configuration to verify reproducibility and consistency.
  • Scenario sets representing low, medium, and high outbreak severity so that the team can compare both normal and stress conditions.

5. Test Timeline

The following timeline shows the planned order of testing activities. A week-based schedule is used so that the plan remains valid even if small project-calendar changes occur. If a phase reveals critical faults, the next dependent phase may be delayed until the critical issue is resolved.

5.1 Test Dependencies

Integration Testing: Successful completion of essential unit tests for connected modules.

System Testing: A stable integrated build with no unresolved critical integration blockers.

Performance Testing: A functionally correct system build that can run end-to-end scenarios.

User Acceptance Testing: A stable system build with acceptable baseline performance.

Beta Testing: Completion of UAT and resolution of major usability blockers.

Bug Fixing and Regression: Defect reports coming from integration, system, performance, UAT, and beta testing.
