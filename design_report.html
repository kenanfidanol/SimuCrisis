1. Introduction

1.1 Purpose of the system

A pandemic simulation system for education called SimuCrisis was created with Unity. Its goal is to assist users in investigating the ways in which pandemic management choices such as lockdowns, vaccination plans, and modifications to healthcare capacity affect the spread of viruses and the burden on healthcare systems. The system is aimed at politicians, researchers, educators, and students.

1.2 Design goals

  • A visual representation of pandemic dynamics that is easy to understand.
  • Availability on common desktop hardware (VR was eliminated because of its low added value).
  • Instantaneous feedback for parameter adjustments and interventions.
  • Architecture that is flexible, maintainable, and modular.
  • Reproducibility through random seeds, run IDs, and scenario preservation.

1.3 Definitions, acronyms, and abbreviations

The susceptible, exposed, infected, recovered epidemiological model is known as SEIR.

Rt: The number of effective reproductions over time.

Scenario: Initial conditions and a saved simulation setup.

Intervention: A policy implemented by users (lockdown, immunization, capacity adjustments).

1.4 Overview

The architecture and key subsystems of SimuCrisis are presented in this High-Level Design Report. Boundary conditions, global control logic, data persistence, hardware/software mapping, and subsystem decomposition are all covered. The whole design is summarized using visual diagrams.

2. Current software architecture

SimuCrisis does not currently have a software architecture. The project was created from the ground up and is intended to be modular for testing and incremental development.

3. Proposed software architecture

3.1 Overview

The modular architecture of SimuCrisis allows the Simulation Core to broadcast snapshots to downstream components while maintaining the model state. The Policy Engine applies user actions that are processed through the User Interface. Time-series indicators, such as Rt, peaks, and load, are compiled by the Metrics Collector. Heatmaps and charts are produced by the Visualization Engine. Export and persistence are managed via Report Generator and Scenario Manager.

3.2 Subsystem decomposition

  • UI Layer (Unity) • Dashboard (charts/heatmap), Policy Panel, Parameter Panel, Scenario Compare
  • Core Layer
  • Data Layer: Simulation Core, Policy Engine, Metrics Collector, Visualization Engine, Scenario Manager, and Report Generator
  • Local storage, configuration/metadata (seeds, provenance), and dataset repository

3.3 Hardware/software mapping

Hardware: A typical desktop computer (no need for virtual reality).

Software: C#, Unity, Python, and local file storage; dataset packs for historical baselines are optional.

3.4 Persistent data management

SimuCrisis stores scenarios and run artifacts locally. Stored artifacts include selected dataset identifiers, parameter sets, intervention schedules, random seeds for reproducibility, and summary statistics. No personal or private health data is stored.

3.5 Access control and security

SimuCrisis runs locally and does not require user authentication in its initial release. Scenario files can be validated (schema checks) before loading to avoid corrupted inputs. Any future online sharing features would require access control and secure transport.

3.6 Global software control

The system follows an event-driven update loop. User actions (start/pause/speed changes, policy application) trigger events that update simulation parameters. Each timestep produces a snapshot that feeds metrics computation and visualization refresh.

3.7 Boundary conditions

Performance may degrade for very large population sizes or dense spatial heatmaps. To mitigate this, the simulation can use aggregated compartmental logic and bounded timestep sizes. Parameter validation prevents unstable configurations (e.g., negative rates, impossible capacities).

4. Subsystem services

Simulation Core:

  • Executes the SEIR epidemiological model to represent the progression of a disease within the simulated population.
  • Inputs: Validated simulation parameters such as infection rate, recovery rate, total population size, simulation time step, and random seed.
  • Outputs: Updated simulation state reflecting population distribution.
  • Error Handling: Validation checks to ensure population values remain within valid ranges.

Policy Engine

  • Service: Manages user interventions and dynamically adjusts simulation parameters.
  • Inputs: Policy Objects and current simulation state.
  • Outputs: Parameter modifiers.
  • Error Handling: Prevents conflicting logic or impossible values.

Metrics Collector

  • Service: Calculates Rt, peak infections, and hospital utilization.
  • Inputs: Simulation snapshots.
  • Outputs: Time-series data and summary statistics.
  • Error Handling: Safe fallback values for unstable calculations.

Visualization Engine

  • Service: Renders charts and spatial heatmaps.
  • Inputs: Metrics and spatial data.
  • Outputs: UI elements.
  • Error Handling: Optimized rendering and fallback handling.

Scenario Manager

  • Service: Saves and loads simulation scenarios.
  • Inputs: User commands and configuration objects.
  • Outputs: Serialized JSON and runtime objects.
  • Error Handling: Schema validation.

Report Generator

  • Service: Compiles final simulation statistics.
  • Inputs: Simulation history and metadata.
  • Outputs: PDF and CSV reports.
  • Error Handling: File system validation.

6. Glossary

Heatmap:
A spatial visualization technique used to represent the intensity or concentration of a specific variable across a geographic or simulated space. In SimuCrisis, heatmaps are primarily used to display infection density and help users visually identify high-risk or heavily affected areas.

Scenario:
A stored simulation configuration that includes initial conditions, disease parameters, and a predefined set of intervention policies. Scenarios allow users to reproduce experiments, compare different strategies, and analyze outcomes under consistent starting assumptions.

Intervention:
A user-defined policy action applied during a simulation run to influence disease dynamics. Examples include lockdown measures, vaccination strategies, and healthcare capacity adjustments. Interventions modify simulation parameters in a controlled manner over time.

Snapshot:
A representation of the complete simulation state captured at a specific timestep. Snapshots include population distributions, calculated metrics, and relevant system variables, and are used for both real-time visualization and post-simulation analysis.
