High Level Design Report

SimuCrisis – Pandemic Disaster Management Simulation

1. Introduction

1.1 Purpose of the System

SimuCrisis is a Unity-based pandemic simulation system developed for educational and decision-support purposes. The system allows users to analyze how pandemic management decisions—such as lockdowns, vaccination strategies, and healthcare capacity adjustments— affect virus spread and healthcare system load.

The system primarily targets policymakers, educators, researchers, and students who wish to gain insight into pandemic dynamics in a safe and controlled environment.

1.2 Design Goals

  • Provide an intuitive visualization of pandemic dynamics
  • Ensure availability on standard desktop hardware
  • Offer real-time feedback for interventions
  • Maintain a modular and extensible architecture
  • Support reproducibility through scenario preservation

1.3 Definitions, Acronyms, and Abbreviations

  • SEIR: Susceptible, Exposed, Infected, Recovered epidemiological model
  • Rt: Effective reproduction number
  • Scenario: A saved simulation configuration
  • Intervention: A policy action applied during a simulation run

1.4 Overview

This document presents the high-level architecture of SimuCrisis, including subsystem decomposition, control mechanisms, data persistence, and boundary conditions. Diagrams are provided to illustrate the overall structure and runtime behavior of the system.

2. Current Software Architecture

SimuCrisis does not rely on a pre-existing software architecture. The system has been designed and implemented from scratch with modularity and incremental development in mind.

3. Proposed Software Architecture

3.1 Overview

The proposed architecture follows a layered and modular design. The Simulation Core maintains the epidemiological model state, while additional components handle policies, metrics, visualization, and persistence.

3.2 Subsystem Decomposition

  • UI Layer: Dashboard, parameter panel, policy panel, scenario comparison
  • Core Layer: Simulation Core, Policy Engine, Metrics Collector, Visualization Engine
  • Data Layer: Dataset repository, scenario storage, configuration metadata

3.3 Hardware / Software Mapping

SimuCrisis is designed to operate on standard desktop computers. The primary technologies used include Unity with C#, optional Python tools for analysis, and local file storage.

3.4 Persistent Data Management

The system stores simulation scenarios and run artifacts locally. Stored data includes parameter sets, intervention schedules, random seeds, and summary statistics. No personal health data is collected or stored.

3.5 Access Control and Security

The initial release of SimuCrisis does not require user authentication. Scenario files are validated before loading to prevent corrupted or incompatible inputs. Future online features would require secure access control.

3.6 Global Software Control

SimuCrisis follows an event-driven control model. User interactions trigger events that update simulation parameters. Each timestep produces a snapshot used for analytics and visual updates.

3.7 Boundary Conditions

Performance limitations may occur for very large population sizes or dense spatial models. To mitigate this, bounded timestep sizes and parameter validation mechanisms are applied.

4. Subsystem Services

Simulation Core

Executes the SEIR model and updates population states at each timestep while ensuring numerical stability.

Policy Engine

Manages user-defined interventions and applies validated parameter modifications.

Metrics Collector

Computes analytical indicators such as Rt, infection peaks, and healthcare utilization.

Visualization Engine

Renders real-time charts and spatial heatmaps representing infection dynamics.

Scenario Manager

Handles saving, loading, and comparison of simulation scenarios.

Report Generator

Exports simulation results in PDF and CSV formats for external analysis.
